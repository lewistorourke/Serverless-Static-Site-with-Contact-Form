# Serverless-Static-Site-with-Contact-Form

# Serverless Contact Form Website

## Project Overview

This project demonstrates the design and implementation of a secure, serverless web application on AWS. A static website is hosted using Amazon S3 and CloudFront, while a contact form is powered by API Gateway, AWS Lambda, Amazon DynamoDB and Amazon SES.

The primary objective was to build a production-style serverless architecture following AWS best practices, including least-privilege IAM, private object storage, Origin Access Control (OAC), defence-in-depth security, and cost optimisation.

# AWS Services Used

| Service | Purpose |
|----------|----------|
| Amazon S3 | Stores the static website files |
| Amazon CloudFront | CDN, HTTPS termination and edge caching |
| Origin Access Control (OAC) | Secure access between CloudFront and S3 |
| Amazon API Gateway | Public API endpoint |
| AWS Lambda | Serverless backend processing |
| Amazon DynamoDB | Stores contact form submissions |
| Amazon SES | Sends email notifications |
| AWS IAM | Least-privilege access control |
| Amazon CloudWatch | Logging and monitoring |

# Build Process

## 1. Private Amazon S3 Bucket

### Purpose

Create secure object storage for the website.

### Key Design Decisions

- Block Public Access left fully enabled.
- Bucket remains private.
- Object Ownership set to Bucket Owner Enforced.
- Versioning enabled.
- Server-side encryption (SSE-S3) enabled.

### Why?

The website only serves static files (HTML, CSS and JavaScript). There is no requirement for compute, operating systems or web servers, making S3 the ideal storage solution.

### Verification

Attempting to access an object directly through its S3 URL returned an **AccessDenied** response, confirming the bucket was not publicly accessible.

---

## 2. Amazon CloudFront

### Purpose

Provide HTTPS, caching and secure public access to the website.

### Key Design Decisions

- Origin Access Control (OAC)
- Redirect HTTP to HTTPS
- GET/HEAD methods only
- Default Root Object configured as `index.html`

### Why?

CloudFront becomes the only service permitted to access the private S3 bucket.

Benefits include:

- Lower latency
- HTTPS support
- Edge caching
- Lower bandwidth costs
- Future integration with AWS WAF and custom domains

### Verification

The website became publicly accessible through the CloudFront domain while remaining inaccessible through the S3 bucket URL.

---

## 3. Amazon DynamoDB

### Purpose

Persist contact form submissions.

### Key Design Decisions

- On-demand capacity mode
- UUID partition key
- Default encryption
- No sort key

### Why?

Contact form traffic is unpredictable and bursty.

Using a UUID guarantees uniqueness while distributing writes evenly across partitions.

### Verification

Successfully created test items and verified them within the DynamoDB console.

---

## 4. AWS Lambda

### Purpose

Process contact form submissions.

### Key Design Decisions

- Python 3.13 runtime
- ARM64 (Graviton)
- Logs-only execution role initially

### Why?

Lambda provides serverless compute that only executes when required.

The execution role intentionally started with only CloudWatch Logs permissions to follow the principle of least privilege.

### Verification

Successfully executed test events and confirmed CloudWatch log creation.

---

## 5. IAM Least Privilege

### Purpose

Allow Lambda to access DynamoDB securely.

### Permissions Granted

- `dynamodb:PutItem`

Only the specific DynamoDB table ARN was authorised.

### Why?

Permissions should only be added when required.

Starting with minimal permissions reduces the attack surface and follows AWS security best practices.

### Verification

Successful DynamoDB writes confirmed both the Lambda code and IAM permissions were correctly configured.

---

## 6. Amazon SES

### Purpose

Send an email notification whenever a contact form submission is received.

### Key Design Decisions

- Verified email identity
- Separate exception handling
- SES Sandbox

### Why?

A failed email should never prevent the user's submission from being stored successfully.

### Verification

Successfully received notification emails generated by the Lambda function.

---

## 7. Amazon API Gateway

### Purpose

Provide a secure HTTPS endpoint for the frontend.

### Key Design Decisions

- HTTP API
- POST `/contact`
- CORS restricted to the CloudFront domain
- Auto Deploy enabled

### Why?

HTTP API provides a lower-cost and simpler alternative to REST API while offering all required functionality.

### Verification

Successfully invoked the endpoint using curl and received valid responses.

---

## 8. Frontend Integration

### Purpose

Connect the static website to the serverless backend.

### Key Design Decisions

- JavaScript Fetch API
- Client-side validation
- Dynamic success/error messages
- CloudFront cache invalidation

### Why?

The frontend submits requests directly to API Gateway without requiring page reloads.

CloudFront serves only the static assets while API Gateway handles all dynamic functionality.

### Verification

Successfully submitted the form through the live CloudFront website.

Confirmed:

- DynamoDB record created
- SES email received
- Success message displayed to the user

# Key Design Decisions

| Decision | Reason |
|-----------|--------|
| S3 instead of EC2 | Static content requires storage, not compute |
| CloudFront instead of Public S3 | HTTPS, caching and secure private bucket access |
| Lambda instead of EC2 | Serverless and pay-per-request |
| DynamoDB instead of RDS | Simple key-value workload with unpredictable traffic |
| HTTP API instead of REST API | Lower cost and sufficient functionality |
| UUID partition key | Prevents collisions and evenly distributes writes |
| On-Demand Capacity | Avoids paying for unused throughput |
| OAC instead of Public Bucket | Keeps S3 completely private |
| Inline IAM Policy | Permissions are unique to this Lambda function |

# Challenges Encountered

During development several issues were intentionally investigated and resolved:

- Default Root Object missing from CloudFront.
- Bucket policy not trusting Origin Access Control.
- CloudFront cache serving stale content.
- Incorrect IAM policy structure.
- API Gateway Invoke URL vs Route ARN confusion.
- SES emails arriving in spam due to sandbox limitations.
- DynamoDB console item count delay.

Each issue provided a deeper understanding of AWS service interactions and common real-world troubleshooting scenarios.

- Differentiating identity-based and resource-based IAM policies.
- Implementing secure error handling and validation.
- Troubleshooting CloudFront caching behaviour.
- Balancing security, performance and cost when designing AWS solutions.
